# MG020 Rescue Postmortem

## Summary

MG020 added structured Codex usage limit detail from the LAX telemetry agent through the hub and into the Glance dashboard. Hub and dashboard support for `limits_detail` landed first, and the dashboard could show `limits_detail_count=2`.

Production rollout then exposed several incorrect assumptions: LAX agent production was not a git worktree, the LAX host did not have npm, the dashboard production directory was Glance config-only rather than an application worktree, and the agent health host port could regress when container and host port names were coupled.

MG020 Rescue deployed agent hotfix `380f392` and hub hotfix `0e9bdbf`. Current remaining limitation: the upstream backend reports limit metadata but not absolute used/reset fields, so the dashboard must show those values as `not reported`.

## Timeline

- Hub deploy first failed because beijing lacked `g++` for a `better-sqlite3` native rebuild under Node 24; installing `g++` fixed `npm ci`.
- Dashboard deploy first failed because the production dir was not a git worktree; the real deploy mode was Glance config-only.
- Agent deploy first failed because the production dir was not a git worktree and the LAX host lacked npm; the real deploy mode was archive plus Docker build.
- Agent rebuild changed host health port to `8081` because the previous compose contract used `HEALTH_PORT` for both container and host mapping.
- Agent MG020 initial payload could leak an `auth.json` marker through sanitized error text.
- Hub `safeTelemetryEvent` did not recursively redact sensitive string markers in historical event payloads.
- Rescue hotfix fixed the agent sanitizer, hub read API redaction, compose port split, and archive deploy script.

## Impact

- Dashboard data proxy degraded during the rollout.
- Monitor `overall_status` became critical while production telemetry was unhealthy.
- No evidence was found in safe logs/output that an actual token was printed, but an `auth.json` marker in an API response was correctly treated as a security failure.
- Public dashboard access remained protected by Authelia.
- ntfy and notify delivery were not changed.

## Root Causes

- Production topology was not documented in the repos.
- Deploy scripts assumed git worktree and npm availability where those assumptions were false.
- Agent error serialization was insufficiently redacted.
- Hub read response redaction was not recursive over strings.
- Health host port and container port were coupled.

## What Went Well

- Smoke checks caught the sensitive marker.
- Dashboard anonymous access remained blocked.
- Backups existed.
- Rescue recovered service without changing ntfy or Authelia.

## What Went Poorly

- Too much diagnosis happened through manual SSH.
- Deployment path discovery happened during the incident instead of being codified.
- State cleanup was attempted before full diagnosis.

## Corrective Actions

- Use archive deploy for LAX.
- Use config-only deploy for the dashboard.
- Keep hub defensive read redaction tests.
- Keep agent recursive sanitizer tests.
- Keep `HEALTH_HOST_PORT` separate from `HEALTH_PORT`.
- Maintain production topology docs.
- Add a production smoke bundle.
- Never manually cat raw state during production diagnosis.

## Non-Goals

- Do not implement notify gateway here.
- Do not migrate old ntfy here.
- Do not delete the old tmux/status chain here.

## Verification

Expected current production baseline:

- Hub smoke: `secret_leak_detected=false`
- Dashboard proxy: `proxy_ok=true`
- Monitor: `overall_status=ok`
- LAX diag: `healthz_18081_ok=true`, marker presence false, `spool_count=0`

`usage_numbers_present_count=0` and `reset_time_present_count=0` are known backend field limitations, not production failures.

## Rollback

- Hub rollback uses the beijing hub backup path printed by the hub deploy script.
- Agent rollback uses the LAX timestamped `backup_path` printed by `deploy-lax-agent-archive.ps1`.
- Dashboard rollback restores the timestamped Glance config backup path from the config-only deploy script.

If a future deploy reintroduces an `auth.json` marker, stop the rollout, preserve the backup, avoid raw state/log output, run safe diagnostics, and fix sanitizer/read redaction before continuing.
